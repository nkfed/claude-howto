## Вхідні дані

```text
$ARGUMENTS
```

Перед початком роботи уточни у користувача (якщо не надано):

- **Епік / Фіча**: до якого епіку належить ця User Story?
- **Роль / Актор**: хто є користувачем? (лікар, пацієнт, адміністратор, система)
- **Мета**: що хоче зробити актор?
- **Цінність**: яку проблему це вирішує або яку вигоду надає?
- **Контекст домену**: медичний, адміністративний, технічний?

**Якщо $ARGUMENTS не порожній** — використай цю інформацію як основу і уточни лише відсутні деталі.

---

## Крок 1: Уточнення вимог

Постав користувачу мінімально необхідні запитання:

1. Хто є основним актором? *(лікар, пацієнт, адміністратор ЕСЗ, зовнішня система)*
2. Яка конкретна дія або функціональність потрібна?
3. Яка бізнес-цінність або очікуваний результат?
4. Чи є обмеження, винятки або особливі умови?

**Зачекай на відповідь перед продовженням.**

---

## Крок 2: Написання User Story

Оформи User Story у стандартному форматі:

```markdown
## [US-XXX] Назва User Story

**As a** [Роль/Актор],
**I want** [дія або функціональність],
**So that** [бізнес-цінність або очікуваний результат].

### Контекст
[Короткий опис контексту — чому це важливо, що передує цій дії]

### Acceptance Criteria

**AC-1: [Назва сценарію]**
- Given [передумова]
- When [дія користувача або системи]
- Then [очікуваний результат]

**AC-2: [Назва сценарію]**
- Given [передумова]
- When [дія]
- Then [результат]

**AC-3: Негативний сценарій / Виняток**
- Given [передумова]
- When [некоректна дія або умова]
- Then [очікувана поведінка системи при помилці]

### Визначення готовності (Definition of Done)
- [ ] Acceptance Criteria підтверджені PO
- [ ] Технічна реалізація погоджена з командою
- [ ] Тест-кейси написані на основі AC
- [ ] Документація оновлена у Confluence

### Залежності
- Пов'язані US: [US-XXX, US-YYY або "відсутні"]
- Блокери: [опиши або "відсутні"]

### Примітки BA
[Додаткові коментарі, відкриті питання, рішення прийняті в ході аналізу]
```

---

## Крок 3: Перевірка якості

Перед фіналізацією перевір:

- [ ] Формат "As a / I want / So that" дотримано
- [ ] Актор — конкретна роль, не "користувач" або "система" узагальнено
- [ ] Мета описує дію, а не рішення (що, а не як)
- [ ] Цінність — бізнес-результат, а не технічний
- [ ] AC покривають: happy path, негативні сценарії, граничні умови
- [ ] AC є тестованими (конкретними, вимірюваними)
- [ ] Немає технічної реалізації в тілі US (лише вимоги)

---

## Крок 4: Презентація та ітерація

1. Покажи готову User Story користувачу
2. Запитай: чи потрібні правки, додаткові AC або уточнення?
3. **Зачекай на підтвердження або зворотній зв'язок**
4. При необхідності — внеси правки та повтори крок 4

---

## Формат виводу

Вивід завжди у **Markdown, готовому для Confluence**.
Мова документації: **українська**.
Технічні терміни (Given/When/Then, Acceptance Criteria, Definition of Done) — залишати **англійською**.

## Приклад

```markdown
## [US-042] Перегляд медичної картки пацієнта лікарем

**As a** лікар загальної практики,
**I want** переглядати зведену медичну картку пацієнта в ЕСЗ,
**So that** я можу прийняти обґрунтоване клінічне рішення без запиту додаткових документів.

### AC-1: Успішний перегляд картки
- Given лікар авторизований і має активну лікарську угоду з пацієнтом
- When лікар відкриває картку пацієнта
- Then система відображає: демографічні дані, діагнози, ліки, алергії, останні аналізи

### AC-2: Відмова в доступі
- Given лікар не має активної угоди з пацієнтом
- When лікар намагається відкрити картку
- Then система повертає помилку 403 з повідомленням "Доступ заборонено"
```
